۱. هدف سامانه

تبدیل transcript جلسات فارسی به خروجی عملیاتی: خلاصه، تسک، پرسش‌وپاسخ RAG، و issueهای Jira — از طریق خط لوله‌ای که برای دموی زنده به‌اندازه‌ی کافی سریع است.

۲. MVP در برابر نسخه کامل (YAGNI)

سند مفهومی اولیه یک سامانه‌ی حداکثری را توصیف می‌کرد. MVP فقط آنچه را که برای اثبات ارزش لازم است نگه می‌دارد و به‌صراحت ثبت می‌کند که چه چیزی به تعویق افتاده — یک موضع منضبط «به آن نیاز نخواهی داشت».

۴. رونویسی صوت (Gemini)

ورودی صوت یک مرحله‌ی جدا پیش از ingest است، تا paste متنی و بارگذاری جلسات نمونه بدون تغییر بمانند. همان GOOGLE_API_KEY و GEMINI_MODEL استفاده می‌شود — بدون سرویس speech-to-text جدا.

مرحلهجزئیات
UIآپلود mp3, wav, m4a, ogg, webm, flac یا دکمه‌ی ضبط (MediaRecorder).
APIPOST /api/transcribe — multipart file ← Gemini generateContent با صوت inline.
خروجیtranscript فارسی با قالب [MM:SS] name: text، قابل ویرایش پیش از تحلیل.
محدودیتحداکثر ۲۰ مگابایت (MAX_AUDIO_BYTES).
پس از رونویسیهمان مسیر POST /api/meetings — تحلیل، Chroma، RAG، Jira.
تفاوت با STT حرفه‌ای: diarization و جریان زنده بخشی از MVP نیستند؛ برای دمو و صوت ضبط‌شده، رونویسی تک‌مرحله‌ای Gemini کافی است.

۷.۱ analysis_agent

from pydantic import BaseModel
from pydantic_ai import Agent

class MeetingAnalysis(BaseModel):
    title: str
    summary: str
    key_points: list[str]
    decisions: list[str]
    tasks: list[MeetingTask]

analysis_agent = Agent(
    'google:gemini-3.5-flash',
    output_type=MeetingAnalysis,
    system_prompt='تحلیل transcript جلسه فارسی. فقط بر اساس متن.',
)

۷.۲ RAG — رفتار هوشمند

  • سلام / hello: بدون جستجو در transcript — پاسخ کوتاهی که به پرسش واقعی درباره‌ی جلسه دعوت می‌کند.
  • پرسش جلسه: Chroma top-4 ← context داخلی برای LLM (transcript خام هرگز در UI نمایش داده نمی‌شود).
  • پاسخ: فارسی طبیعی با نسبت‌دادن («علی گفت…») و در صورت نیاز یک footnote کوتاه.
# تشخیص small talk
is_small_talk("hello") -> chitchat_agent, sources=[]

# پرسش واقعی
chunks = chroma.query(meeting_id, question)
rag_agent.run(context + question) -> RagAnswer

۷.۳ facilitation_agent

پس از ingest، برگزارکننده می‌تواند پیشنهادهای بهبود جلسه‌ی بعد را ببیند — با لحن مربی‌گری، نه سرزنش.

  • ورودی: transcript، MeetingAnalysis، آمار مشارکت گویندگان، نوع جلسه.
  • خروجی: FacilitationReport — نکات مثبت، بهبودها، دستور کار جلسه‌ی بعد، پیشنهاد timebox، خلاصه‌ی مربی‌گری، و امتیاز اختیاری ۱ تا ۵ برگزارکننده.
  • API: GET /api/meetings/{id}/facilitation
  • UI: تب «راهنمای جلسه» در صفحه‌ی جلسه (lazy load).
ایجاد Jira از یک route مستقیم API است. issueها به انگلیسی (title_en, context_en) نوشته می‌شوند تا نمایش چپ‌به‌راست در Jira تمیز باشد، در حالی که UI فارسی می‌ماند.

قواعد RTL/LTR:

  • html lang="fa" dir="rtl" — متن فارسی راست‌چین.
  • کد، متغیر محیطی، URL و نام API در dir="ltr" قرار می‌گیرند.
  • نمودارهای Mermaid در یک wrapper با جهت LTR رندر می‌شوند.

صفحه‌ی /meeting/[id] داده را روی سرور Astro از FastAPI می‌گیرد (نه فقط با JS سمت کلاینت)، که مشکل «گیرکردن روی در حال بارگذاری» را رفع و first paint را بهتر می‌کند.

اسکرین‌شات‌ها با ./scripts/capture-ui-screenshots.sh (Playwright) تولید می‌شوند — backend و frontend باید در حال اجرا باشند.

۱۳. اجرای سامانه

کل پشته با یک اسکریپت اجرا می‌شود (پورت‌ها را آزاد می‌کند، هر دو سرور را بالا می‌آورد، و هنگام خروج پاک‌سازی می‌کند):

chmod +x scripts/run-dev.sh
./scripts/run-dev.sh
# باز کنید: http://localhost:4321

یا دو سرویس را جدا اجرا کنید:

./scripts/run-backend.sh    # FastAPI روی :8000
./scripts/run-frontend.sh   # Astro روی :4321

۱۵. مدیریت خطا

خطارفتار
timeout در Geminiیک‌بار retry + پیام فارسی
transcript بدون speakerبرچسب [ناشناس]
Jira 401راهنمای بررسی .env
فایل صوتی نامعتبر / بزرگHTTP 400 (فارسی) — فرمت یا MAX_AUDIO_BYTES
Gemini billing (403 dunning)پیام فارسی راهنمای billing برای رونویسی و تحلیل
RAG بدون match«در این جلسه یافت نشد»

یک سرویس اختصاصی gemini_errors، استثناهای ارائه‌دهنده (ModelHTTPError و خطاهای httpx) را به HTTPExceptionهای فارسی و کاربرپسند نگاشت می‌کند، تا یک hold در billing یا کلید نامعتبر هرگز به‌صورت یک stack trace خام ظاهر نشود.

اسپرینت‌های بعدی: به سند ۳ مراجعه کنید — RAG چندجلسه + آرشیو full-text (Sprint 2)، نگاشت assignee + OAuth در Jira (Sprint 3)، auth/workspace + متادیتای تقویم (Sprint 4)، PostgreSQL + queue + استقرار (Sprint 5)، و قابلیت‌های چندعامله (Sprint 6+). STT جریانی زنده عمداً در نقشه‌ی راه نیست — رونویسی Gemini نیاز MVP را پوشش می‌دهد.
